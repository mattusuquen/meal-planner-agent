# Design Document: AI Food Tracking & Planning Agent

**Version:** 1.1
**Author:** Matt
**Status:** Implemented (v1)

---

## 1. Overview

An AI-powered meal planning and nutrition tracking application. Users generate personalized meal plans and recipes via an AI agent, log meals against verified nutrition data, and monitor progress through recalculating dashboards. The core differentiator vs. generic AI tools: **nutrition accuracy backed by a real food database**, not LLM-estimated macros.

### Goals
- Generate meal plans and recipes tailored to user goals, preferences, and dietary restrictions
- Guarantee macro/calorie accuracy by grounding all nutrition data in the USDA FoodData Central database
- Provide progress dashboards that recalculate automatically as user data changes
- Enhance recipes with AI-generated images
- Act as an agent: proactively adjust plans based on logged data and progress trends

### Non-Goals (v1)
- Barcode scanning
- Wearable/fitness integrations
- Social features
- Native mobile apps (responsive web first)

## 3. Feature Set

### 3.1 Onboarding & Profile
- Goal selection: lose / maintain / gain weight, target rate (lbs/week)
- Biometrics: age, sex, height (ft/in), weight (lbs), activity level
- TDEE via Mifflin-St Jeor + activity multiplier; macro split 30% protein / 45% carbs / 25% fat
- Dietary preferences: cuisines, restrictions, allergies, disliked ingredients
- Meal structure: 3 meals or 3 + snacks
- Onboarding gate: middleware checks `profiles.onboarding_complete`; only redirects to `/onboarding` on first login, never on subsequent logins
- Profile saved via `upsert` so it works regardless of whether the trigger fired at signup

### 3.2 AI Meal Planning
- Weekly plan generated by GPT-4o-mini with profile macros as system context; `response_format: json_object`
- Day keys are local-date ISO strings (`YYYY-MM-DD` via `toLocaleDateString("en-CA")`) to avoid UTC-midnight timezone bugs
- Plan stored as JSONB in `meal_plans`; old plans archived (status = `archived`) before new plan saved
- Single-meal swap: GPT generates replacement within the remaining daily macro budget
- Grocery list: aggregates `recipe_ingredients` from plan's recipe IDs, categorizes by keyword matching (Produce / Proteins / Dairy / Pantry / Grains / Other)
- Plan fetch in journal modal uses same local-date week-start calculation to match stored keys

### 3.3 Recipe System
- **Generation pipeline:** GPT-4o-mini structured output (Zod schema) → each ingredient matched to USDA → macros summed from verified data → saved to `recipes` + `recipe_ingredients`
- LLM provides only ingredient names and gram weights; never supplies final nutrition numbers
- Macro totals accumulated by direct reduce (no intermediate per-100g conversion to avoid divide-by-zero)
- Nutrition cache (`nutrition_cache`) stores USDA results by `fdcId`; written via service-role client to bypass RLS
- Recipe image generated fire-and-forget via `POST /api/recipes/[id]/image` after recipe save; uploaded to `recipe-images` Storage bucket; recipe page polls for the image URL every 5 seconds for up to 60 seconds

### 3.4 Food Logging
- Six entry methods: plan (one-tap from today's meal plan), recipe, USDA search, AI text parse, photo, quick-add
- Entry method stored as `entry_method` on `logged_meals`; used for accuracy weighting in future agent reviews
- After every write (INSERT or DELETE), `recalcDailyTotals` recomputes the `daily_totals` row server-side (not via DB trigger)
- Errors in `recalcDailyTotals` are caught and logged but do not fail the overall request — the meal insert still returns 200
- All POST routes upsert a profile row (`ignoreDuplicates: true`) before inserting FK-dependent rows, handling users who signed up before the migration trigger was in place

#### USDA Search
- Search returns results with `fdcId`, `description`, `calories/protein_g/carbs_g/fat_g` per 100g
- Log entry is quantity-in-grams based (not servings), calculated from `per_100g` × grams / 100

#### Photo Logging
- Photo uploaded to `meal-photos` (private bucket) via service-role client
- Signed URL (1hr) passed to GPT-4o-mini vision; permanent URL returned to client for diary display
- Each detected item: name + estimated grams → USDA match → verified macros
- Confirmation-first UX: editable draft with confidence badges before logging

#### Text / AI Parse
- GPT-4o-mini parses free-text description into structured items (name, quantity, unit, estimated grams)
- Each item USDA-matched; same confirmation draft UI as photo

### 3.5 Progress Dashboards
- **Daily:** macro rings vs. targets, calorie budget bar, today's logged meals list; data from `/api/dashboard/daily`
- **Weekly/Monthly:** trend charts (Recharts), adherence rate, avg calories/protein; data from `/api/dashboard/trends`
- **Progress:** weight vs. calorie intake dual-axis chart; weight stats (current, total change, weekly rate)
- All reads from `daily_totals` and `weight_entries` — never sum raw logs on page load

### 3.6 Weight Tracking
- Weight entered in lbs; stored in kg (`weight_lbs × 0.453592`); displayed back as lbs
- `upsert` on `(user_id, entry_date)` — one entry per day, re-submitting overwrites
- Weekly rate: 7-day rolling average vs. prior 7-day average (lbs/week)

## 4. Data Model

```sql
profiles (
  id uuid pk references auth.users,
  goal text, activity_level text,
  height_cm numeric, birth_date date, sex text,
  calorie_target int, protein_g int, carbs_g int, fat_g int,
  dietary_restrictions text[], allergies text[], dislikes text[], cuisines text[],
  meal_structure text default '3_plus_snacks',
  display_name text,
  onboarding_complete boolean default false
)

recipes (
  id uuid pk, created_by uuid references profiles,
  name text, instructions jsonb, servings int,
  image_url text, source text check (source in ('ai','user')),
  cuisine text, prep_minutes int,
  -- denormalized verified totals per serving:
  calories numeric, protein_g numeric, carbs_g numeric, fat_g numeric
)

recipe_ingredients (
  id uuid pk, recipe_id uuid references recipes on delete cascade,
  usda_fdc_id int references nutrition_cache,
  raw_text text, quantity numeric, unit text, grams numeric
)

nutrition_cache (
  usda_fdc_id int pk, description text,
  per_100g jsonb,  -- { calories, protein_g, carbs_g, fat_g }
  fetched_at timestamptz
)

meal_plans (
  id uuid pk, user_id uuid, week_start date,
  plan jsonb,   -- { "YYYY-MM-DD": { "Breakfast": MealCell | null, ... } }
  status text   -- 'active' | 'archived'
)

logged_meals (
  id uuid pk, user_id uuid, logged_date date,
  recipe_id uuid null, custom_entry jsonb null,
  servings numeric default 1, meal_slot text,
  entry_method text check (entry_method in ('plan','recipe','search','text','photo','quick')),
  photo_url text null
)

daily_totals (
  user_id uuid, date date,
  calories numeric, protein_g numeric, carbs_g numeric, fat_g numeric,
  meals_logged int,
  primary key (user_id, date)
)

weight_entries (
  id uuid pk, user_id uuid,
  entry_date date, weight_kg numeric,
  unique (user_id, entry_date)
)
```

RLS enabled on all user-owned tables (`user_id = auth.uid()` or `created_by = auth.uid()`). `nutrition_cache` is publicly readable; written only via service-role client.

Auto-create profile trigger on `auth.users` INSERT — but all POST API routes also upsert the profile row defensively to handle users created before the trigger existed.

## 5. Recalculation Architecture

Daily totals are recomputed server-side (not via DB triggers) on every log write:

1. **Write-time recomputation.** Every `POST /api/log` and `DELETE /api/log` calls `recalcDailyTotals(supabase, userId, date)` after the write succeeds.
2. **`recalcDailyTotals`** fetches all `logged_meals` for that user+date, sums macros from `custom_entry` JSONB or from joined `recipes` row, then upserts `daily_totals` on conflict `(user_id, date)`.
3. **Target changes don't rewrite history.** Targets are read from `profiles` at render time and compared against stored totals.
4. **No DB triggers in v1.** Simpler to reason about; trigger-based cascade is a v2 option if write throughput requires it.

## 6. AI Agent Architecture

```
User request → Next.js API route handler
  ├── Plan generation:   GPT-4o-mini, json_object mode, Zod validation, retry once on parse failure
  ├── Meal swap:         GPT-4o-mini, remaining daily budget passed as context
  ├── Recipe generation: GPT-4o-mini, Zod schema → USDA match loop → macro reduce → save
  ├── Text parse:        GPT-4o-mini → structured items → USDA match
  ├── Photo analysis:    GPT-4o-mini vision (signed URL) → structured items → USDA match
  ├── Ingredient matcher: USDA /foods/search → top result → cache in nutrition_cache
  ├── Nutrition calc:    Pure TS functions in lib/nutrition.ts (no LLM involvement)
  └── Image generation:  gpt-image-1, fire-and-forget fetch after recipe save
```

- `response_format: { type: "json_object" }` on all generation calls; Zod validates before saving
- Plan generation retries once on Zod parse failure before returning 500
- Image generation is non-blocking; recipe page polls `GET /api/recipes/[id]/image` every 5s

## 9. Known Design Decisions & Gotchas

| Decision | Rationale |
|---|---|
| `toLocaleDateString("en-CA")` for all date keys | `toISOString().split("T")[0]` returns UTC date; parsing a date string without time as UTC midnight causes off-by-one day errors in non-UTC timezones |
| Service-role client for `nutrition_cache` | RLS blocks anon writes; nutrition cache is global (not per-user) so service-role is appropriate |
| Fire-and-forget image generation | Image generation takes 10-30s; blocking the recipe save on it degrades UX significantly |
| `recalcDailyTotals` errors non-fatal | A failed recalculation shouldn't roll back a successful meal log; dashboard will self-correct on next load |
| Profile upsert on every POST | Handles users created before migration trigger; idempotent with `ignoreDuplicates: true` |
| `ignoreDuplicates: true` on profile upsert | Prevents overwriting existing profile fields when only `id` is provided |

## 10. Setup Requirements

1. Supabase project with URL + anon key + service-role key in `.env.local`
2. Run `supabase/migrations/001_initial_schema.sql` in Supabase SQL Editor
3. Create Storage buckets: `recipe-images` (public), `meal-photos` (private)
4. OpenAI API key with access to `gpt-4o-mini` and `gpt-image-1`
5. USDA FoodData Central API key (free at api.nal.usda.gov)
6. If users signed up before running the migration, run: `INSERT INTO public.profiles (id) SELECT id FROM auth.users ON CONFLICT (id) DO NOTHING;`

## 11. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Ingredient matching errors | USDA top-result matching; confirmation-first draft UX before logging |
| USDA API latency/downtime | `nutrition_cache` table; matched results cached indefinitely |
| LLM plan misses macro targets | Post-generation Zod validation; retry once; tolerance visible in plan UI |
| Image generation cost | Generated once per recipe, cached in Storage; no regeneration unless explicitly triggered |
| Photo portion estimates inaccurate | Confirmation draft with per-item confidence badges; macros always from USDA, never from vision model |
| Timezone date key mismatch | All date strings use `toLocaleDateString("en-CA")` consistently across client and server |
| Missing profile row (FK violation) | All write routes upsert profile defensively before insert |
